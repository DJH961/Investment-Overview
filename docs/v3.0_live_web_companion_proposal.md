# v3.0 — Live Web Companion (GitHub Pages) — Proposal

> **Status:** proposal / for sign-off. Nothing in here is built yet. This
> document is the plan of record for a browser-based, on-the-go companion that
> shows **live** portfolio stats — total and per holding — computed in the
> browser against **live prices**, with the secret ledger published as an
> **encrypted** blob so it is safe to expose on a public GitHub repository.
>
> Companion to [`docs/mobile_android_app_proposal.md`](mobile_android_app_proposal.md)
> (which it effectively supersedes for the "check on the go" use case) and
> [`docs/v2.0_split_cloud_security_plan.md`](v2.0_split_cloud_security_plan.md).

---

## 1. Vision

Keep the **desktop app exactly as it is** — the powerful local tool for CSV/XLSX
imports, exports, deep analysis, and the authoritative ledger. Add a **thin,
read-only web companion** hosted on **GitHub Pages** that you can open on *any*
device (phone, work laptop, a friend's machine) with nothing but a URL and a
passphrase, and immediately see:

- **How the portfolio is doing right now** — total value, total/▲today gain,
  XIRR, YTD/MTD/daily growth, expense drag, dividend yield — recomputed against
  **live market prices**, not a stale export.
- **How each holding is doing right now** — shares, current price, **today's
  move (€ and %)**, market value, portfolio weight, unrealised P/L, and
  per-holding XIRR — so you can see, at a glance, *which holdings are driving
  today's movement* and the current market impact on the whole portfolio.

### Why this is possible (the two things we proved)

1. **A browser can fetch your live prices for free.** Verified by direct test:
   **Twelve Data** returns live quotes with the HTTP header
   `Access-Control-Allow-Origin: *` (the permission a webpage needs to call an
   API). **Frankfurter** supplies the EUR/USD rate, free, keyless, CORS-open.
   Yahoo (the desktop's source) and Stooq are **not** browser-usable — the web
   build simply uses a different price tap.
2. **A browser can do the maths.** Every metric you care about live (XIRR, TWR,
   CAGR, growth %, per-holding returns) is pure arithmetic over a small list of
   dated cashflows plus one "value right now" number. The desktop already proves
   this in `domain/returns.py`; we re-implement those leaf functions in the web
   language and **guard them with a parity test suite** (§6).

### The single hard truth, stated honestly

Your **Fidelity mutual funds** (`FXAIX`, `FSKAX`, `FSELX`) and the money-market
fund (`VMFXX`) have **no intraday price** — mutual funds strike one NAV per day
after close. "Live" for those means *latest daily NAV*. Your **ETFs** (`VTI`,
`SCHK`, `DAX`/Global X) **do** tick live. This is a property of the asset class,
not a limitation of this design — the desktop app (yfinance) has the identical
ceiling. The UI will label NAV-only holdings so the distinction is explicit.

### What stays where

| Concern | Desktop app (private, unchanged) | Web companion (new, public Pages) |
|---|---|---|
| Imports, editing, onboarding, settings | ✅ sole owner | ❌ never |
| Deep analysis, audit export, dev tools | ✅ | ❌ |
| Authoritative ledger (SQLite tiers) | ✅ | ❌ (only a minimized, encrypted extract) |
| Live "how am I doing" view | ✅ | ✅ (the whole point) |
| Writes of any kind | ✅ | ❌ read-only |

## 2. Architecture at a glance

```
┌──────────────────────────────────────────────────────────────┐
│ DESKTOP APP (private, on your machine — unchanged behaviour)  │
│                                                              │
│  import / edit ──► ledger (SQLite tiers)                     │
│                         │                                    │
│        readmodels.build_mobile_export()  ← NEW (minimized)   │
│                         │                                    │
│        publish_service:  encrypt (AES-GCM)  ── NEW           │
│                         │  push (GitHub API, PAT)            │
└─────────────────────────┼────────────────────────────────────┘
                          ▼
        GitHub repo (PUBLIC after the §7 scrub)
          • /web              ← static front-end (public, harmless)
          • portfolio.enc     ← encrypted blob (release asset, overwritten)
                          │  served by GitHub Pages
                          ▼
┌──────────────────────────────────────────────────────────────┐
│ BROWSER (any device)                                         │
│  1. load app shell from Pages                                │
│  2. ask passphrase → WebCrypto decrypts portfolio.enc        │
│     (decrypted data lives in memory only)                    │
│  3. fetch live quotes (Twelve Data) + FX (Frankfurter)       │
│  4. compute KPIs + per-holding stats (ported domain maths)   │
│  5. render dashboard                                         │
└──────────────────────────────────────────────────────────────┘
```

**Trust boundary:** GitHub only ever holds public front-end code + opaque
ciphertext. Plaintext figures exist only in your browser's memory, after you
type the passphrase, on a device you control. The API key never touches the
repo (it is entered once per device — §5.4).

## 3. The data contract — the minimized, encrypted export

A new builder, `readmodels/build_mobile_export()`, produces a compact JSON
document (then encrypted — §4). It is deliberately split into **two classes** of
data, which is the key design decision that determines what is "live" vs
"as-of-export".

### 3.A Live inputs — recomputed in the browser against live prices

These are small primitives; the browser combines them with live prices/FX.

- `meta`: `schema_version`, `app_version`, `generated_at`, `as_of`,
  `base_currency: "EUR"`, available display currencies.
- `holdings[]`, one per open position (from `services/positions_service.
  compute_positions`, so split-adjustment & cost basis stay in Python):
  - `symbol`, `name`, `asset_class`, `broker`, `account`, `native_currency`
  - `shares` (split-adjusted as of `as_of`)
  - `cost_basis_native`, `cumulative_dividends_cash_native`
  - `price_symbol` — the **Twelve Data ticker** to quote (may differ from the
    ledger symbol; see §5.3)
  - `price_type` — `"market"` (ETF → live intraday) or `"nav"` (mutual/money
    market → daily NAV; browser shows last NAV, no intraday move)
  - `last_known_price_native` — fallback if the live quote is missing/NAV stale
  - `cashflows[]` — dated, signed cashflows for **this holding** (for
    per-holding XIRR), using the `domain/returns.py` sign convention
    (contributions negative; sells/dividends/interest positive).
- `portfolio_cashflows[]` — dated, signed cashflows for the **whole portfolio**
  (for portfolio XIRR/TWR).
- `cash`: non-market balances that still count toward total value —
  the Direct savings (Tagesgeld) balance and broker settlement cash
  (`compute_cash_balance`), each with currency. These are added to total value
  as-is (no live price needed).
- `period_openings` — to recompute the **current** month & year live (§3.C):
  - `month_start_value_eur`, `year_start_value_eur` (total **and** per holding)
  - within-current-period flows are already derivable from the cashflow lists by
    date, so no duplication is needed.

### 3.B Precomputed read-models — displayed as-of-export (not recomputed live)

These depend on long **price history** that is impractical to fetch live; the
desktop already computes them, so they ride along and are shown with a clear
"as of <generated_at>" stamp.

- `periods.completed_monthly[]`, `periods.completed_yearly[]` — every month/year
  **except the current one** (the current period is recomputed live — §3.C).
- `analytics` bundle — equity curve, attribution, and all risk metrics that need
  a daily series: `annualized_volatility`, `sharpe_ratio`, `sortino_ratio`,
  `max_drawdown`, `beta`, `alpha`, `calmar_ratio`, `ulcer_index`,
  `historical_var`, `historical_cvar`, `skewness`, `excess_kurtosis`,
  best/worst month, monthly win rate.
- `deposits` summary.
- `transactions` ledger — **optional**, off by default (it is the most sensitive
  and the bulkiest section). A setting controls whether to include it.

### 3.C Live recompute of the current period (your requested refinement)

The current month and current year are recomputed **live** because the browser
has everything it needs without price history:

```
current_period_growth_pct =
    f(opening_value_eur,            // exported (start of month/year)
      live_closing_value_eur,       // computed now from live prices + FX
      within_period_net_flows)      // derived from exported cashflows by date
```

The formula mirrors `_period_query.aggregate`'s `growth_pct` exactly, and is
covered by the parity suite (§6) so the live current-period number agrees with
what the desktop would show. Completed prior periods stay frozen as exported.

### 3.D Precision & types

All decimals are serialized as **strings** (e.g. `"1234.500000"`), matching the
existing snapshot contract, and parsed in the browser with **decimal.js** to
avoid float drift. Dates are ISO-8601. `None` → `null`. Bump `schema_version`
on any breaking shape change.

## 4. Encryption — safe on a public repo

The export is encrypted **on your machine** before it ever leaves it.

- **Cipher:** AES-256-**GCM** (authenticated — detects tampering as well as
  hiding contents). Implemented with the `cryptography` package on the Python
  side, decrypted with the browser-native **WebCrypto** `SubtleCrypto` API.
- **Key derivation (decided):** from your dedicated **mobile passphrase**
  (separate from the SQLCipher DB passphrase) via a deliberately slow KDF.
  *A KDF (key-derivation function) just turns your passphrase into the actual
  encryption key while being intentionally slow, so an attacker who grabs the
  public blob can only test a few guesses per second instead of billions.*
  **Locked in for v3.0: PBKDF2-HMAC-SHA256** at **600,000 iterations** —
  browser-native (WebCrypto), zero JS dependencies, nothing to decide. A future
  upgrade to **Argon2id** (memory-hard, even tougher against cracking rigs) is
  noted as optional and is *not* required for launch.
- **Envelope** (the published `portfolio.enc`, base64 fields):
  `{ v, kdf, kdf_params{salt, iterations|argon_params}, nonce, ciphertext, tag }`.
  Salt and nonce are random per publish.

### Threat model — read this before going public

The encrypted blob is **public ciphertext**. Treat it as permanently downloadable
and attackable offline, forever. Therefore:

- **The passphrase + KDF cost is the entire wall.** Use a long, unique
  passphrase (a 5–6 word passphrase is far stronger than a "complex" short one).
  Never reuse it.
- **AES-GCM** guarantees nobody can silently corrupt the blob into misbehaving.
- **History hygiene:** publishing the blob as a **release asset that is
  overwritten** each time (recommended) keeps ciphertext **out of git history**,
  so old encrypted snapshots do not accumulate forever. If you ever instead
  commit the blob into the tree, a future passphrase compromise retroactively
  exposes every historical blob in history — so prefer the release-asset route.
- **Data minimisation:** the export already strips to shares + cashflows +
  derived numbers. Do **not** add raw account numbers, counterparties, or notes.

## 5. Desktop (Python) — what changes

All additions are **additive**; default behaviour is unchanged when the feature
is off.

### 5.1 New: `readmodels/mobile_export.py`
`build_mobile_export(session, *, include_transactions=False) -> dict` assembling
§3.A + §3.B by reusing the existing compute (`positions_service`,
`metrics_service`, `periods` aggregate, `analytics.build`). No new business
logic — same single-source-of-truth discipline as the existing snapshot.

### 5.2 New: `services/publish_service.py`
Orchestrates **export → encrypt → push**:
- `export()` → `build_mobile_export()`.
- `encrypt(payload, passphrase)` → AES-GCM envelope (reuse/extend
  `storage/encryption.py`).
- `publish(envelope)` → upload to GitHub. **Recommended transport:** the GitHub
  REST API release-asset endpoint (overwrite the single asset on a fixed
  `live-data` release tag) using a **fine-grained Personal Access Token (PAT)**.
  - *What a PAT is, plainly:* a revocable, scope-limited password GitHub issues
    so a program can act for you. We scope it to **only this one repo** with
    **only "Contents" write**, and store it in the **OS keyring** (same place the
    SQLCipher passphrase lives) — never in code, `.env`, or the repo. If it ever
    leaks, you revoke that one token; nothing else is affected.

### 5.3 Price-symbol mapping & coverage
`price_symbol` / `price_type` per holding (§3.A). One open task before launch:
with a free Twelve Data key, **confirm coverage** of the awkward tickers —
`FXAIX`, `FSKAX`, `FSELX` (mutual funds), `VMFXX` (money market), and `DAX`
(Global X, which collides with the index name and may need an exchange
qualifier). Any symbol Twelve Data can't serve falls back to
`last_known_price_native` and is flagged NAV-only. Finnhub/FMP are documented
fallbacks (both CORS-open) for ETF symbols if needed.

### 5.4 Triggers (auto-publish)
- **After every successful import** — hook in `services/importer_service` post
  commit.
- **On graceful app close** — via `shutdown.py` (`app.on_shutdown`) /
  `request_shutdown`.
- **Manual "Publish now"** button + a "last published at" indicator.
All three are gated by a master toggle. (Each publish is one export+encrypt+one
HTTPS PUT — cheap.)

### 5.5 New settings (`config.py`, `INV_DASHBOARD_` prefix)
- `publish_enabled: bool = False`
- `publish_repo: str | None` (e.g. `DJH961/Investment-Overview`)
- `publish_release_tag: str = "live-data"`
- `publish_token: str | None` (normally from keyring, env for CI only)
- `mobile_passphrase: str | None` (normally from keyring)
- `publish_include_transactions: bool = False`

### 5.6 Settings UI panel — "Live web companion"
Token + passphrase capture (written to keyring), repo field, include-transactions
toggle, master enable, **Publish now**, last-published timestamp, and a coverage
check button that quotes each `price_symbol` once and reports gaps.

### 5.7 New CLI — `inv-dashboard-publish-web`
Mirror of `inv-dashboard-export-snapshot` for manual/scheduled publishing and
for CI smoke tests. Honours `--refresh`.

### 5.8 New tool — `tools/gen_parity_vectors.py`
Dumps golden input/output vectors from the **Python** domain functions to
`tests/parity/vectors.json` (the JS parity suite consumes these — §6).

## 6. The web app — what gets built, and the parity test suite

### 6.1 Stack (chosen; happy to revisit)
- **TypeScript** + **Vite** (tiny static build; output to `/docs` or a
  `gh-pages` branch). TypeScript gives the parity suite real types for the
  ported maths.
- **decimal.js** — decimal arithmetic mirroring Python `Decimal`.
- **ECharts** — charts (treemap allocation, equity curve, per-holding bars).
  Light, capable, no account needed.
- **WebCrypto** (native) — decryption.
- **Vitest** — unit + parity tests.
- **PWA** (manifest + service worker) — installable, offline app shell. The
  service worker caches **only** the app shell and the **encrypted** blob; the
  **decrypted** data is held in memory only and never cached/persisted.

### 6.2 Modules
- `crypto.ts` — PBKDF2/Argon2id key derivation + AES-GCM decrypt of the envelope.
- `priceClient.ts` — batched Twelve Data `/quote` (one call, comma-separated
  symbols; gives `price`, `previous_close`, `percent_change` for **today's
  move**) + Frankfurter FX. Caching + graceful degradation to
  `last_known_price_native`.
- `compute/` — the **ported domain maths**: `xirr`, `twr`, `cagr`,
  `annualizeReturn`, `totalGrowthPct`, `capitalGain`, `yearsBetween`,
  `totalGrowthPctCompounded`, plus the position roll-up (value = shares × price ×
  FX) and the current-period recompute (§3.C).
- `view/` — pages (§6.4).
- `setup.ts` — first-run **setup screen on the website** (your point 5): enter
  Twelve Data API key (stored in this browser's `localStorage`, never in the
  repo). **Passphrase convenience (decided):** offer **"remember passphrase on
  this device"** (opt-in, stored in `localStorage`), and where the device/browser
  supports it, gate that stored passphrase behind a **biometric / platform
  unlock** via the **WebAuthn** platform authenticator (Face ID / Touch ID /
  Windows Hello). If WebAuthn isn't available, fall back to typing the passphrase.
  The passphrase still never leaves the device and is never sent to the repo.

### 6.3 Parity test suite (the guarantee that JS == Python)
Two layers, both run in CI:

1. **Leaf-function parity.** `tools/gen_parity_vectors.py` runs the *real* Python
   `domain/returns.py` (and the current-period formula) over a battery of
   fixtures — ordinary portfolios, single cashflow, all-same-day, negative XIRR,
   leap-year spans, zero-cost-basis, multi-currency — and writes inputs +
   **expected outputs** to `tests/parity/vectors.json`. Vitest feeds the **same**
   inputs to the TS implementations and asserts equality within tolerance
   (money ≤ 1e-6; rates/percentages ≤ 1e-8; XIRR ≤ 1e-6, since both sides use an
   iterative solver).
2. **End-to-end pipeline parity.** A fixed `mobile_export` fixture + a **frozen**
   set of prices/FX is fed to **both** the Python overview/period compute and the
   TS live compute; every KPI and per-holding figure must match. This guards the
   whole chain, not just the leaf maths.

**CI wiring** (`.github/workflows`): a job regenerates `vectors.json` from Python
and **fails if it differs from the committed copy** — so any Python change that
alters a number forces a deliberate regen — then runs the Vitest parity suite.
Python and JS therefore cannot silently drift.

### 6.4 Pages/features — what's easy, what's hard, what we build

| Web view | Source | Effort | Build in v3.0? |
|---|---|---|---|
| **Overview** — KPIs (value, total/▲today gain, XIRR, YTD/MTD/daily, expense, yield) + allocation treemap | **Live** | Easy | ✅ **Hero** |
| **Per-holding** — shares, price, **today's move €/%**, value, weight, unrealised P/L, per-holding XIRR/YTD | **Live** | Easy–Med | ✅ **Hero** |
| **Monthly / Yearly** — prior periods frozen; **current month & year live** | Mixed (§3.C) | Med | ✅ |
| **Calculator / Projection** — forward growth from current live value | **Live** | Easy | ✅ |
| **Deposits** — contributions/dividends/interest summary | Export | Easy | ✅ |
| **Analytics / Risk** — Sharpe, Sortino, drawdown, vol, beta/alpha, VaR, equity curve, attribution | Export (history-bound) | Med–Hard | ✅ **display-only**, stamped "as of export" |
| **Transactions ledger** — full row list | Export (optional, sensitive) | Easy to render | ⚠️ behind unlock, off by default |
| Imports / editing / onboarding / settings / audit / dev tools | — | — | ❌ **never** (desktop is sole writer) |

*Why analytics is display-only:* its inputs (`risk.py` / `risk_extras.py`) need a
**daily value/return series** spanning years. Fetching that live would be many
API calls and slow; recomputing intraday adds no value (risk stats don't move
meaningfully tick-to-tick). So they are computed on the desktop and shown
as-of-export. If live history is ever wanted, it's a clean follow-up (Twelve Data
`time_series`), explicitly out of scope here.

## 7. Going PUBLIC — mandatory security scrub (do this BEFORE flipping visibility)

You chose to keep everything in **one repo** and make it public. The decisive
risk: **making a repo public exposes its entire git *history*, not just the
current files.** Deleting sensitive files in a new commit does **not** remove
them from history.

### 7.1 Real financial data that was in the repo — anonymized + history rewrite (decided)
`docs/Comparison Files/` previously contained your **actual** transactions and
holdings (the Fidelity `2023–2026.csv` exports, the three `audit-export-*.json`
snapshots, `customActivityReport.xlsx`, and `Investments.xlsx`).

**Decision: keep one repo and rewrite history** (no fresh repo). Concretely:
- The real exports were **replaced in the working tree with anonymized synthetic
  fixtures** — same public symbols (plus a few similar holdings added), same
  action types (incl. the SCHK share distribution), values scaled drastically,
  and all external-bank / transfer descriptions scrubbed. The snapshots stay
  internally consistent (value = shares × price; %/XIRR/ratios untouched). See
  `docs/Comparison Files/README.md` for the per-file detail.
- **`Investments.xlsx` is flagged as un-fakeable and removed** — its 7
  formula-driven sheets can't be faithfully anonymized; keep it locally only.
- The real blobs are **purged from git history** with `git filter-repo` against
  the `docs/Comparison Files/` path, then force-pushed. A `git bundle` backup is
  taken first. Note GitHub may cache old objects briefly — treat anything ever
  pushed as potentially seen and rely on the fact that only scaled, PII-free
  fakes remain.

Add `audit-export-*.json` and `*.xlsx` patterns to `.gitignore` (while keeping
the anonymized CSV/JSON fixtures tracked) so real exports can't be re-added.

### 7.2 Secrets — never in the repo
- Ensure `.gitignore` covers `.env` (and confirm no committed `.env`).
- The **PAT** and **mobile passphrase** live **only** in the OS keyring — never
  in code, `.env`, CI logs, or commits.
- Audit `.github/workflows/*` so no token is ever echoed; use repo **Secrets**
  for any CI token, never inline.

### 7.3 Personal identifiers in docs/code/comments — review & scrub
Going public turns README/comments into public PII. Review at least:
- **Accessibility note — keep explicit (decided):** the design targets
  **colorblind-safe (red-green / deuteranopia-protanopia) palettes**. This is an
  intentional **accessibility** decision and is to be **kept stated explicitly**
  throughout docs and UI — do **not** scrub or genericize it.
- **Named accounts/vendors:** README/spec name **"Direct Savings Bank
  Tagesgeldkonto"** and specific brokers/holdings. Decide if you want your real
  bank and portfolio composition publicly tied to you; genericize if not.
- **Local paths / usernames** in docs or examples (e.g. `%USERPROFILE%` is fine;
  a real machine username is not).
- **Fixtures:** `tests/adapters/fixtures/*` use a fake account number
  (`12345678`) — fine; just confirm no real IDs crept into any fixture.

### 7.4 In-code guard rails to add
- A header comment in `mobile_export.py` / `publish_service.py`: *"This repo is
  PUBLIC. Never log, commit, or persist decrypted data, tokens, or
  passphrases."*
- Logging review: ensure the publish path logs only non-sensitive metadata
  (counts, timestamps), never payload contents, the token, or the passphrase.
- A `SECURITY.md` documenting the encrypted-blob model and the
  "ciphertext is public, passphrase is everything" assumption.

### 7.5 Pre-flip checklist
- [ ] Sensitive data files removed **and** purged from history (or fresh repo).
- [ ] `.gitignore` blocks data files, `.env`, build artefacts.
- [ ] No secrets in code, history, or workflows; PAT + passphrase keyring-only.
- [ ] PII in docs/comments reviewed and scrubbed to taste.
- [ ] Encrypted blob published as an **overwritten release asset** (not committed).
- [ ] Strong, unique mobile passphrase set; KDF cost tuned.
- [ ] `SECURITY.md` added.

## 8. Resolved decisions (no open questions)

Every prior decision is now locked; nothing here is awaiting input.

1. **One repo, not two.** Keep desktop + web companion together; manage the
   public-visibility change via the §7 scrub (history rewrite + anonymized
   fixtures) rather than a separate repo.
2. **Auto-publish triggers:** after every successful import **and** on graceful
   app close, **plus** a manual **"Publish now"** button. Each trigger is
   individually toggleable.
3. **Push credentials:** a **fine-grained PAT scoped to this one repo**, stored
   in the OS keyring.
4. **Two separate passphrases:** a dedicated **mobile passphrase** for the web
   blob, distinct from the SQLCipher DB passphrase.
5. **KDF:** **PBKDF2-HMAC-SHA256 @ 600k iterations** (decided, §4). Argon2id is a
   future optional upgrade only.
6. **Price API key:** entered on the website's **setup screen**, kept in browser
   `localStorage`, never in the repo (§6.2).
7. **Current period recomputes live:** the in-progress month and year are
   recomputed against live prices; completed periods stay frozen as exported
   (§3.C).
8. **Transactions ledger:** **off by default** in the export; can be enabled
   behind the unlock if ever wanted (§3.B).
9. **Passphrase convenience:** opt-in **"remember on this device"** with
   **biometric/WebAuthn** gating where supported, falling back to typing (§6.2).
10. **Accessibility note kept explicit:** colorblind-safe palettes are stated
    deliberately and are **not** scrubbed (§7.3).
11. **History remediation:** **rewrite history** of the existing repo with
    `git filter-repo`; real comparison data already replaced by anonymized
    synthetic fixtures, `Investments.xlsx` removed as un-fakeable (§7.1).

The only remaining *verification* (not a decision) is the **Phase 0 coverage
spike**: confirm Twelve Data serves the awkward symbols
(`FXAIX`/`FSKAX`/`FSELX`/`VMFXX`/`DAX`) with a free key before building. ETFs and
FX are already confirmed.
