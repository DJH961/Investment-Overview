# v2.10.1 — closing the historic spreadsheet ↔ dashboard gap

This plan follows up on **v2.10.0**, which reconciled the developer audit export
(`docs/Comparison Files/audit-export-2026-05-31 (1).json`) with the authoritative
`docs/Comparison Files/Investments.xlsx` on four valuation defects (Fidelity
share distributions, dividend double-counting, the withdrawal sign convention,
and the historic leading-gap backfill). After those fixes the **current** value
matches the spreadsheet almost exactly, but the **historic** balances and the
growth rates derived from them remained materially different. This document
records the investigation, ships one robustness feature, and scopes the actual
fix for the remaining gap.

---

## 1. What v2.10.0 already fixed (do not redo)

So we build on, rather than re-litigate, the prior release:

- **Fidelity `Type=Shares` distributions** reclassified from `dividend_cash` to
  `split` (migration `0007`); `SCHK` resolves to its real share count.
- **Single dividend-income definition** — reinvested legs valued once, cash legs
  not double-counted; income vs realized cash separated.
- **Withdrawal sign convention** netted once across deposits/period queries.
- **Historic leading-gap backfill** — price/FX refresh now refetches from the
  earliest *needed* date, so early months are no longer valued at zero because
  the cache started late.
- **Early non-reinvested dividends** recognised as income.

Contributions, dividends and the *current* portfolio value now line up with the
spreadsheet to within rounding. The residual is purely in the **historic
holdings valuation**.

---

## 2. The symptom, measured

Comparing the spreadsheet `Growth` sheet (monthly closing balance, USD) against
the v2.10 audit export's `monthly.closing_value_display`:

| Month | Spreadsheet (USD) | Dashboard (USD) | Gap |
|---|---:|---:|---:|
| 2023-03 | 24,134.97 | 22,172.77 | **−8.1 %** |
| 2024-09 | 39,827.80 | 35,789.65 | **−10.1 %** |
| 2024-10 | 39,339.53 | 36,019.74 | **−8.4 %** |
| 2026-03 | 47,745.61 | 45,052.64 | **−5.6 %** |
| 2026-04 | 52,123.56 | 52,135.83 | **+0.0 %** |
| 2026-05 | 54,582.02 | 54,594.32 | **+0.0 %** |

The dashboard understates every historic month by ~8–10 %, then **snaps to
parity in 2026-04**. Crucially there are two discrete *steps* where the gap
shrinks:

- **2024-10**: −10.1 % → −8.4 % (a ~$0.7k correction)
- **2026-04**: −5.6 % → 0.0 % (a ~$2.7k correction)

The current value matches because it is priced from the live feed; only the
*past* is wrong.

---

## 3. Root cause — split back-adjustment mismatch

The two steps line up exactly with the only two `split` transactions in the
ledger:

| Split | Date | Share delta |
|---|---|---:|
| SCHK 2:1 | 2024-10-11 | +26.10 |
| VUG | 2026-04-21 | +37.16 (7.43 → 44.59) |

The mechanism:

1. **yfinance returns split-adjusted (back-adjusted) closes.** After a split it
   rewrites *all* prior history downward by the split ratio, so a 2023 price is
   already divided by the factor of every split that happened since.
2. **The ledger applies a split as a share-count change on the split date**
   (`positions_service.compute_positions` adds the split `quantity` only from
   that date forward).
3. For any date **before** a split, the dashboard therefore multiplies the
   *pre-split* share count by the *post-split-adjusted* (smaller) price —
   understating that holding by the split ratio.

So in 2023 both SCHK and VUG are understated (each carries a future split that
back-adjusts its price). At 2024-10 the SCHK share count catches up to its
adjusted price → step one. At 2026-04 VUG catches up → step two and full parity.
This fully accounts for the ~8–10 % historic understatement; no other holding,
FX, dividend or contribution discrepancy is needed to explain the data. The
user's instinct ("old price data for some funds is off") is correct — the prices
are not literally `0`, they are *split-adjusted while the share counts are not*.

---

## 4. Shipped in this release — corrupt-price-history flag

The user also asked to flag, in the UI, any instrument whose price data is
"completely inaccurate" (e.g. a feed that returns `0`). This is a real, separate
failure mode (yfinance occasionally returns `0.0` for a missing print, which then
forward-fills through `close_as_of`), and it is now surfaced:

- `prices_repo.instrument_ids_with_nonpositive_close` detects any cached close
  `≤ 0`.
- `prices_service.instruments_with_price_anomalies` wraps it tier-aware (reads
  the cache DB under split-DB layouts).
- The Overview shows a red banner naming the affected symbols (distinct from the
  existing "held but valued at zero *today*" banner), and the audit export
  carries `price_data_warning` per position for offline reconciliation.

This does **not** fix the split issue (those prices are non-zero); it makes the
*other* "old price is wrong" failure mode visible so it can never again silently
understate history.

## 5. Plan — fix the split back-adjustment (the actual gap)

The goal: a historical valuation on date `d` must use a share count and a price
that are on the **same adjustment basis**. Options, with trade-offs:

### Option A — value pre-split dates with split-adjusted share counts *(recommended)*
Keep yfinance's adjusted prices (they are the simplest to fetch) and, when
valuing a past date, scale the share count by the product of every split ratio
that occurs **after** `d`. Concretely:

- Add a helper that, for an instrument, returns the cumulative split factor
  applied after a given date (derived from the `split` ledger rows — ratio =
  `(shares_after / shares_before)` at each split date).
- In `positions_service.compute_positions` (historical branch) and the period /
  equity-curve queries, multiply the as-of share count by that factor before
  pricing with the adjusted close.
- Today's valuation is unchanged (no future splits ⇒ factor 1).

Pros: no extra network calls, prices stay consistent with the live feed.
Cons: needs the split ratio, not just the share delta — store/derive it.

### Option B — store and value with *unadjusted* prices
Fetch `auto_adjust=False` (raw) closes so a 2023 price is the real 2023 price,
matching the pre-split share count directly.

Pros: conceptually simplest (share count and price are both "as-traded").
Cons: requires a price-cache refetch/rebuild and changes the meaning of cached
closes everywhere (latest_close, daily growth, benchmarks); higher blast radius.

### Option C — back-adjust the *ledger* share counts instead
Rewrite historical share counts to the post-split basis (as brokerages do on
statements).

Pros: ledger then matches adjusted prices natively.
Cons: mutates immutable transaction history; conflicts with the cost-basis and
the imported broker data; not reversible. **Not recommended.**

**Recommendation: Option A.** It is the smallest, most localised change, keeps
the cached prices and the live path untouched, and is straightforward to unit
test (a holding with a known split should value the same fraction of spreadsheet
truth before and after the split date).
