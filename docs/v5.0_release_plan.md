# v5.0 Release Plan — the "make a complete app even better" jump

_Drafted 2026-06-28 against `v4.16.2`._

This is a curated, evidence-backed backlog for a **v5.0 major version jump**. It
was produced by a fresh full-tree audit of both halves of the product — the
Python NiceGUI desktop app (`src/investment_dashboard/`, the source of truth) and
the `web/` mobile companion (the Vite/TS PWA that springboards off the desktop's
exported snapshot).

The codebase is already mature: `docs/maintenance_audit.md` §0–§5 is essentially
cleared, the desktop is on par or superior to the web build on the eight Tiingo
issue classes (`docs/python_desktop_parity_plan.md`), and CI is green. So v5.0 is
**not** a rescue release — it is a deliberate quality and capability jump that
closes the few remaining real gaps and raises the engineering floor.

Every item below cites concrete code. During the audit, three plausible-sounding
"bugs" were investigated and **rejected as already-correct**, so they are *not*
in this plan: `web/src/prices.ts::convert` already guards `fromRate.isZero()`
(line 581); `web/src/compute.ts` already null-checks `valuePrevEur` before the
today-move percentage (line 953); and `positions_service.py`'s
`native_rate != 0` check (line 309) already rejects `Decimal('0.00')`. The plan
sticks to gaps that survived verification.

---

## Headline theme for v5.0

> **"One brain, both screens, fully trustworthy numbers."**
> Close the mobile-companion feature gap so the phone is a first-class client of
> the desktop, harden the shared financial math and its failure modes so every
> displayed number is either correct or honestly blank, and raise the
> engineering floor (type safety, CI breadth, supply-chain integrity) to match
> the maturity of the product.

The work is grouped into six workstreams (A–F). A–C are the user-visible jump;
D–F are the durability that justifies a major-version number. Suggested
sequencing is in the last section.

---

## Workstream A — Mobile companion feature parity (the user-facing jump)

The desktop exposes **12+ pages** (`ui/pages/`: overview, holdings, transactions,
monthly, yearly, analytics, projection, calculator, deposits, diagnostics,
settings, help). The web companion exposes **4 tabs** — Overview, Periods, Risk,
Calculator (`web/src/ui.ts:1455-1458`). The gap below is the single biggest
lever for "make it even better on the device people actually carry."

- **A1 — Transactions / ledger view on mobile.** The export already carries the
  ledger opaquely (`mobile_export.py` `include_transactions` flag;
  `web/src/types.ts:316` `transactions?: unknown`, with the comment at line 278
  noting it "rides along… carried opaquely") but the web UI never renders it.
  Add a Transactions tab: a searchable, currency-aware buy/sell/dividend list so
  a user can audit individual cashflows on the phone. _Highest-value gap._

- **A2 — Per-holding drill-down.** Desktop has a dedicated holdings page
  (`ui/pages/holdings.py`) with per-security P/L, cost-basis cross-check and
  history; web shows holdings only as Overview rows (`web/src/ui.ts` overview
  panel). Add a tap-through detail sheet per holding (cost basis, unrealised
  P/L, today's move, per-currency growth, mini price spark).

- **A3 — Data-health surface on mobile.** Desktop has a Diagnostics page
  (`ui/pages/diagnostics.py`: FX gaps, stale prices, provider status). The web
  build silently degrades (missing FX → blank legs) with no health screen. Add a
  lightweight "Data health" panel surfacing snapshot age (`generated_at`,
  consumed in `compute.ts`), per-symbol freshness (already tracked in
  `freshness.ts`/`holding-status.ts`), and free-tier credit budget
  (`cache.ts` credit log) so the user can see *why* a number looks off.

- **A4 — In-app help / metric glossary.** Desktop ships `ui/pages/help.py` plus a
  rich tooltip copy module (`ui/copy/tooltips.py`); web has scattered tooltips
  but no guide. Port the metric glossary so first-time mobile users understand
  XIRR/TWR/Sharpe etc. without the desktop.

- **A5 — Settings parity (read + selected writes).** Web settings
  (`web/src/ui.ts:1430`) is essentially API-key + lock config. Desktop settings
  (`settings.py`, 2,109 lines) manages accounts, instruments, allocations,
  benchmark, risk-free symbol and manual refresh. v5.0 should at minimum surface
  **read-only** account/instrument/allocation/benchmark config on mobile and a
  manual "refresh now" control; full write-back is a stretch goal (and overlaps
  the long-standing "multi-device write queue" backlog item in
  `docs/maintenance_audit.md §3.2`).

- **A6 — Period drill-down depth.** Desktop monthly/yearly pages
  (`ui/pages/monthly.py`, `yearly.py`) expose per-period holding-level
  contribution; web's Periods tab (`renderPeriodsPanel`) shows period rows only.
  Add an expand-to-holding-contribution view per period.

## Workstream B — Financial-math correctness & honesty

These are the numbers users trust against their broker statements. None are
crashes; they are precision and "silently-wrong-vs-honestly-blank" issues.

- **B1 — XIRR / CAGR float round-trips inside a Decimal codebase.** The desktop
  XIRR Newton–Raphson and the annualisation helpers drop to `float()` mid-Decimal
  (`domain/returns.py:64,75` Newton iterations; `:227,251,336`
  `Decimal(repr(float(base) ** float(exponent)))`). Over 15–20 year ledgers this
  can drift a basis point or two from a pure-Decimal result — visible to a user
  reconciling against a spreadsheet, which is exactly this app's stated parity
  goal (`docs/spreadsheet_parity_comparison.md`). Evaluate a higher-precision
  power (e.g. `Decimal`-context `exp/ln`, or `mpmath`) and lock the result with a
  golden-vector test. _Note: the web port (`web/src/returns.ts`) must move in
  lockstep or the phone and desktop XIRR will disagree — see B6._

- **B2 — Cost-basis fallback silently collapses EUR/USD growth on old exports.**
  When an export predates the per-trade-date `cost_basis_eur` field,
  `web/src/compute.ts:868-871` falls back to converting native cost at *today's*
  spot, which makes EUR, USD and native growth collapse to the same number
  (value also uses today's spot). The desktop already emits the correct field
  (`mobile_export.py`); v5.0 should (a) bump `schema_version` and have the web
  build **detect and badge** a legacy blob ("growth shown in native only —
  re-publish for currency split") rather than silently showing wrong numbers.

- **B3 — Money-market detection leans on heuristics for old blobs.** The export
  now carries explicit `is_money_market` (`mobile_export.py`), but
  `web/src/money-market.ts` still falls back to a ticker/name heuristic when the
  flag is absent — a new settlement fund ISIN or a renamed fund bypasses it,
  gets sent to the price provider (wastes a free-tier credit) and returns a stale
  $1.00. Tie the fallback to the same schema-version gate as B2 and add the
  known-symbol list to the export so the heuristic is data-driven, not hardcoded.

- **B4 — Average-cost edge case on a sell-before-buy ledger.**
  `positions_service.py` only updates average cost when `shares_before > 0`; a
  corrupted/misordered ledger whose first leg is a sell silently drops the
  proceeds from the cost basis forever. Decide the policy explicitly (reject the
  row in import validation, or surface it in Diagnostics) rather than silently
  skipping. Add a unit test for the misordered-ledger case.

- **B5 — Split-factor handling lacks dedicated edge-case tests.** Historical
  valuation re-derives share counts through `cumulative_split_factor`s
  (`positions_service.py`, `splits_repo`). There is no focused test for multiple
  splits on one instrument, a split exactly on the `as_of` boundary, or a
  corrupt (zero/negative) factor — yet a reversed ratio would silently corrupt
  *every* historical point. Add a golden split-scenario test alongside the
  existing `tests/services/test_metrics_golden_master.py`.

- **B6 — Lock the desktop↔web math contract with shared vectors.** The web
  re-implements XIRR, TWR, growth, money-market and the projection annuity
  (`web/src/returns.ts`, `compute.ts`, `phase4.ts` — the latter explicitly
  "so the web calculator agrees with the desktop one"). Today nothing fails CI if
  they drift. Add a committed fixture of canonical inputs→outputs generated by
  the Python side and assert it from **both** test suites, so any divergence in
  B1/B2/B3 (or future edits) is caught at PR time.

## Workstream C — Robustness, empty/error/offline states & accessibility

The app degrades quietly today; v5.0 should degrade *legibly*.

- **C1 — Distinguish "empty portfolio" from "load/sync failed."** Desktop
  overview renders the same `empty_state()` whether the ledger is genuinely empty
  or a refresh failed (`ui/pages/overview.py`, `_overview_query.py`). A user who
  sees "no holdings" after a failed refresh may re-import and duplicate rows. Add
  a distinct error/stale state on both desktop and web.

- **C2 — Honest, recoverable config-corruption path on web.** `config.ts`
  JSON-parse failures throw a generic "isn't valid JSON" with no recovery action.
  Given the secret-store/IndexedDB design (`secret-store.ts`), a partial write
  leaves the user stuck. Add an "import backup / reset & re-pair" recovery UI.

- **C3 — Offline-first PWA fallback.** `web/src/pwa.ts` registers a service
  worker, but there is no explicit "you're offline, showing last snapshot from
  {time}" banner and no guaranteed last-good-snapshot render path. Make offline a
  designed state, not a blank screen — this is a mobile-first app
  (`web/README.md` design priority).

- **C4 — Accessibility pass (WCAG 2.1 A/AA).** Concrete gaps found: Settings
  inputs lack `<label>`/`aria-label` (`web/src/ui.ts` settings form), the
  market-status `role="status"` element has no `aria-live` so refresh/staleness
  changes aren't announced, and the spinner is a glyph with no semantic label.
  Audit and fix labels, live regions, focus order and contrast across both UIs.

- **C5 — Event-listener lifecycle on lock/unlock.** Auto-lock adds activity
  listeners with `{ passive: true }` but removes them without matching options
  (`web/src/auto-lock.ts` / `app.ts`); long sessions can accumulate orphaned
  listeners. Normalise add/remove option objects and add a teardown test.

- **C6 — FX staleness should be visible, not silent.** Missing/zero FX legs
  already degrade to blank (good), but a *stale-but-present* cached EUR/USD rate
  (`web/src/cache.ts` `readCachedEurUsd`) is used without an age badge, unlike
  quotes which carry an `at` timestamp. Stamp and surface FX age the same way.

## Workstream D — Engineering quality: type safety & CI breadth

- **D1 — Extend mypy beyond `domain/`.** CI type-checks only
  `src/investment_dashboard/domain` (`.github/workflows/ci.yml`,
  `.pre-commit-config.yaml`); the adapters, repositories, services, readmodels,
  api and ui layers — the large majority of ~42k LOC — have no static type gate.
  Ratchet mypy outward layer-by-layer (services and readmodels first, where the
  financial logic lives), each layer made strict only once clean, so debt can't
  silently grow.

- **D2 — CI OS / runtime matrix.** Both CI jobs run `ubuntu-latest` only, Python
  pinned to 3.12, Node pinned to 22 (`ci.yml`). `pyproject.toml` already declares
  3.13 support and the primary *user* platform is Windows (installer, file-locking
  quirks). Add a Windows runner and a Python 3.13 cell at minimum.

- **D3 — Validate & publish the web build artifact.** The web job runs
  `npm run build` but never uploads or smoke-tests `dist/` (`ci.yml`), and the
  Pages deploy (`pages.yml`) is the only consumer. Add an artifact upload + a
  minimal headless-load smoke check and assert the built `__APP_VERSION__`
  matches `package.json` (which already must match `pyproject.toml`).

- **D4 — Cross-layer contract test (web ↔ export schema).** The encrypted
  snapshot is the integration seam (`readmodels/mobile_export.py` ↔
  `web/src/compute.ts`/`types.ts`). A Python schema change can silently break the
  phone. Add a test that the Python export validates against the same schema the
  web `types.ts` expects (codegen or a shared JSON-schema), and round-trips a
  Python-encrypted blob through the browser-crypto decrypt path. This also pins
  the `schema_version` negotiation that B2/B3 rely on.

## Workstream E — Supply-chain & release integrity

- **E1 — Verify downloaded interpreter & pip in the installer.** `installer/
  bootstrap.py` downloads the CPython embeddable and `get-pip.py` over HTTPS via
  `urlopen` (`:78,91,128`) with **no SHA-256/manifest verification**. The
  installer is the trust boundary for a financial app; pin and verify each
  artifact hash against a committed manifest. _(CodeQL/security-review should be
  run on any change here.)_

- **E2 — Verify the self-update download.** The launcher self-updates from the
  GitHub Releases API (`installer/launcher.py:164`) without integrity/signature
  verification. Add hash (and ideally signature) verification of the downloaded
  release asset before it replaces the running install.

- **E3 — Dependency vulnerability scanning.** There is no `dependabot.yml` and no
  `pip-audit`/`npm audit` step in CI (`.github/` has only `workflows/`). Add a
  scheduled dependency-audit job and Dependabot for `uv.lock` and
  `web/package-lock.json` — material for a money app shipping a desktop binary.

## Workstream F — Maintainability & docs hygiene

- **F1 — Break up the three monoliths.** `web/src/app.ts` (7,944 lines) and
  `web/src/ui.ts` (4,435) carry the entire state machine and all rendering;
  `ui/pages/settings.py` (2,109) mixes accounts, instruments, allocations,
  refresh and reset in one file. None is tested in isolation
  (`app.ts`/`ui.ts` have almost no direct unit tests). Split by concern behind
  stable interfaces and add focused tests as each piece comes out — this is the
  prerequisite that makes Workstream A/C changes safe.

- **F2 — Documentation single-source-of-truth.** `docs/` holds ~25 plan/proposal
  files, several superseded (multiple Tiingo plans, pre-v3 audits). Keep
  `docs/maintenance_audit.md` as the live backlog, move completed/obsolete plans
  under `docs/history/` (as the v2.x plans already are), and add a short
  `docs/README.md` index marking each doc current vs historical. Wire the
  proposed "audit-freshness" check (`maintenance_audit.md §4.2`) so the backlog
  can't drift more than one release behind `pyproject.toml`.

- **F3 — Structured, correlatable logs + crash digest in the support bundle.**
  Logging is human-readable text only (`services/logging_service.py`). Add an
  optional structured (JSON) sink and a per-operation correlation id so a single
  "refresh → FX → metrics" flow can be traced, and fold an error digest into the
  existing Data Health support bundle for faster single-user debugging.

## Original proposals (beyond the audit findings)

Two ideas the audit did not surface but that fit the v5.0 "even better" framing:

- **O1 — A single end-to-end "trust" test that boots the desktop on a seeded
  ledger, publishes a snapshot, and renders it in the web build, asserting the
  headline numbers match.** Today desktop and web are tested in isolation; this
  one test would catch *any* export/compute/crypto/schema drift between the two
  halves in a single signal, and is the natural capstone of D4 + B6. It is the
  highest-leverage single addition for a "one brain, both screens" release.

- **O2 — A "reconciliation mode" that ingests the broker's own period-end
  statement value and shows the delta against the app's computed close.** The
  repo already keeps broker comparison fixtures
  (`docs/Comparison Files/`, `docs/spreadsheet_parity_comparison.md`); turning
  that one-off audit into an in-app feature (paste/enter the broker's stated
  total, see the reconciled difference and which holding/FX leg explains it)
  directly serves this app's core promise of trustworthy, statement-matching
  numbers and gives users a self-serve way to catch the very B-class issues above.

## Explicit non-goals for v5.0 (unchanged from existing caveats)

Carried verbatim from `docs/maintenance_audit.md §3.3` — do **not** scope-creep
into these: real-time intraday quotes beyond the existing live tip, multi-user
or real auth, tax-lot/capital-gains accounting, trade execution, or a Savings
Bank CSV importer (no broker export exists).

## Suggested sequencing

1. **Foundation first (F1, D1, B6, D4, O1):** split the monoliths and stand up
   the cross-layer contract + golden math vectors *before* touching numbers, so
   every later change is guarded.
2. **Correctness (B1–B5):** land the math hardening behind the new vectors.
3. **Parity & UX (A1–A6, C1–C6):** build the mobile features and legible
   failure states on the now-safe foundation.
4. **Floor-raising in parallel (D2, D3, E1–E3, F2, F3):** CI matrix, artifact
   validation and supply-chain integrity can proceed independently throughout.
5. **Capstone (O2):** reconciliation mode as the flagship v5.0 feature once the
   numbers are trustworthy.

Pick items into `docs/maintenance_audit.md` as they're scheduled so the live
backlog stays the single source of truth.
