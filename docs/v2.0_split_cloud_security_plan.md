**Status:** planning document for the v2.0 release line. Supersedes the
"canonical + cache" two-file sketch (see "Correction" below) with the
three-tier partition that is already partially shipped on
`copilot/prepare-multi-device-database-sync`.

deliberately does not preclude it (Savings-style write queue on the
config tier, conflict-free choices on the ledger tier), but no
cross-device coordination ships in v2.0.

## 2. Correction vs. the initial two-file sketch

The initial plan proposed a **two-file** split (`canonical` + `cache`)
and lumped `app_config` and target allocations onto the canonical
(ledger) file. That was wrong for two reasons:

1. **Different sync semantics.** `app_config` and `target_allocations`
   are *user preferences*, not facts that happened. They change
   independently of the ledger, they are the natural source of conflicts
   in any future multi-device write path, and on a shared cloud folder
   they should be writable from a secondary device via a queue while the
   ledger stays single-writer. Mixing them with `transactions` on one
   file forces the same single-writer + immutability discipline on the
   wrong rows.
2. **Different encryption / sync surface.** The ledger contains the
   long-lived holdings record we most want to protect; the config tier
   needs to sync but is much smaller and lower-stakes; the cache is
   regenerable and **must not** sync (it would just churn bytes through
   the cloud folder all day). One key per surface is cleaner than one
   key with three different policies.

v2.0 therefore commits to the **three-tier** split already encoded in
the ORM:

| Tier     | File              | Contents                                                                                  | Cloud-synced? | Encrypted at rest? | Writer model            |
|----------|-------------------|-------------------------------------------------------------------------------------------|---------------|--------------------|-------------------------|
| Ledger   | `ledger.sqlite`   | `accounts`, `instruments` (incl. `expense_ratio`†), `transactions`                        | Yes           | Yes                | Single-writer (primary) |
| Config   | `config.sqlite`   | `app_config`, `target_allocations`, `target_allocation_items`, `instrument_overrides`*    | Yes           | Yes                | Single-writer in v2.0; queue-ready schema |
| Cache    | `cache.sqlite`    | `price_history`, `fx_history`, `position_snapshots`, `price_cache_metadata`               | **No**        | No                 | Per-device, local only  |

\* `instrument_overrides` is a Phase 1b deliverable — see §3.1.

† **Correction (v2.0 user note):** `expense_ratio` is an intrinsic
property of the security (Vanguard publishes VTI's TER; the user does
not choose it). It stays on the ledger-tier `Instrument`, alongside
`symbol`, `name`, `asset_class`, and `native_currency`. The Settings UI
field that lets the user type it in is a *manual enrichment* workaround
for the fact that we do not yet ingest it automatically from yfinance —
not a user preference. A future enhancement (out of scope here) will
auto-populate it from a market-data adapter and downgrade the Settings
field to a read-only display with a "refresh" button.

## 3. Part A — Finish the DB split

### 3.1 Phase 1b — peel mutable instrument fields into the config tier

`models/instrument.py` calls out that `category`, `expense_ratio`,
`target_weight_pct`, and `active` are "temporarily retained" on the
ledger-tier `Instrument`. Two of those are genuine config-tier
preferences (user-chosen grouping label, user-toggled visibility); one
is dead duplication; one is an intrinsic security fact that was
mis-grouped with the others:

| Field               | Decision         | Reason                                                                                       |
|---------------------|------------------|----------------------------------------------------------------------------------------------|
| `category`          | Move to config   | User-chosen grouping label ("Total US", "Growth"); changes independently of the security.    |
| `active`            | Move to config   | User-chosen visibility toggle.                                                               |
| `target_weight_pct` | **Drop**         | Dead code — every read path uses `target_allocations` / `target_allocation_items` instead.   |
| `expense_ratio`     | **Stay on ledger** | Intrinsic property of the ETF (Vanguard sets VTI's TER). The Settings UI editor is manual enrichment, not a preference. |

Deliverable:

- New `ConfigBase` model `InstrumentOverride` keyed by `instrument_id`
  (loose reference; integer, indexed), with columns `category`,
  `active`.
- Repo: `instrument_overrides_repo` with `get_for(instrument_id)`,
  `get_many(ids)`, `upsert(instrument_id, **fields)`,
  `set_active(instrument_id, bool)`. Default values when no row exists:
  `category=None`, `active=True`.
- Service: read paths (`positions_service`, `_overview_query`,
  Settings UI listing) compose the ledger `Instrument` with its override
  at the service boundary, so call sites stay shaped like today.
- Migration `0004_split_instrument_overrides` — create
  `instrument_overrides`; copy current `(instrument_id, category,
  active)` rows; drop `category`, `active`, and the unused
  `target_weight_pct` columns from `instruments` (batch mode).
- Update `instruments_repo.update_instrument` to no longer accept
  `category` / `active` (they route through `instrument_overrides_repo`
  instead); keep `name`, `asset_class`, `expense_ratio` on the ledger
  path.
- Tests: round-trip through repo, override defaults when row absent,
  Settings dialog edits land on the right table, expense-ratio edits
  still land on the ledger.

This lands **before** Phase 2 (engine split). Otherwise the engine
split ships with config-tier data still living on the ledger file.

### 3.2 Phase 2 — one engine per tier

Goal: move each `MetaData` onto its own SQLite file without touching
any business code outside `db.py`, `boot.py`, and the alembic config.

**`db.py`:**
- Replace the single `_engine` / `_SessionLocal` with three of each:
  `ledger_engine` / `LedgerSession`, `config_engine` / `ConfigSession`,
  `cache_engine` / `CacheSession`. The SQLite pragma listener stays on
  all three.
- Add three explicit context managers: `ledger_session_scope()`,
  `config_session_scope()`, `cache_session_scope()`.
- Keep `session_scope()` as a back-compat alias that yields the ledger
  session and log-warns once at first use, so call-site migration can be
  staged.
- Add `bind_routing` / a custom `Session.get_bind` so an accidental
  cross-tier query (e.g. joining `PriceHistory` to `Instrument`) raises
  immediately instead of silently failing or going to the wrong file.

**Call-site migration:**

| Module                        | Tier(s)             |
|-------------------------------|---------------------|
| `accounts_repo`, `instruments_repo`, `transactions_repo`, `importer_service` | ledger only |
| `app_config_repo`, `allocations_repo`, `display_currency_service`, `instrument_overrides_repo` | config only |
| `prices_repo`, `price_cache_repo`, `fx_repo`, `snapshots_repo`, `fx_service` | cache only |
| `prices_service.refresh_prices` / `refresh_due_prices` | ledger (read instruments) + cache (write) |
| `snapshots_service.get_or_compute` | ledger + config + cache |
| `metrics_service`, `positions_service` | ledger + config + cache |
| `boot._refresh_fx`, `boot._refresh_prices` | ledger + cache |

Multi-tier services open multiple sessions inside one function;
**write order is always ledger → config → cache**, each committed before
the next opens, because cache failures are recoverable and config
failures must not leave the ledger half-written.

**Alembic:**
- Split into three migration trees: `migrations/ledger/`,
  `migrations/config/`, `migrations/cache/`, each with its own `env.py`
  and `versions/`.
- `alembic.ini` gains three sections (`[alembic:ledger]`,
  `[alembic:config]`, `[alembic:cache]`) selected via `-n`.
- Each tree gets a synthetic `0001_split` migration that creates the
  tier's tables in their post-split shape; the historical `0001`–`0004`
  chain stays for users on a single-file install (see §3.3).
- `boot._run_migrations()` upgrades all three to head in order
  ledger → config → cache.
- `render_as_batch=True` everywhere for SQLite-safe ALTERs.

### 3.3 Phase 2 data migration (one-shot)

`python -m investment_dashboard.tools.split_db --from PATH/to/db.sqlite`:

1. Detect schema version of the source file. Refuse to run unless it is
   at the pre-split head (`0004`).
2. Compute target paths from current `Settings` (ledger path =
   configured ledger path; config and cache go next to it unless
   overridden — see Part B).
3. For each target tier:
   - Create the new file via `tier_base.metadata.create_all`.
   - Copy rows table-by-table inside a single transaction.
   - `PRAGMA integrity_check` post-copy.
   - Stamp the new alembic head for that tier.
4. Rename source to `db.sqlite.pre-split.bak` (do not delete).
5. Verify the post-copy row counts match per table; abort and restore
   on mismatch.

The tool runs offline (no NiceGUI, no network), takes `--dry-run`, and
exits non-zero on any check failure. `boot._run_migrations` detects the
legacy single-file layout and prints a one-line instruction pointing at
this tool rather than auto-running it (we don't want a silent rewrite of
the user's data file).

### 3.4 Cache-orphan janitor

`price_history.py` already mentions a "boot-time cache-orphan janitor".
Implementation:

- `cleanup_orphan_cache_rows(ledger_session, cache_session)` —
  `SELECT id FROM instruments`, then `DELETE FROM price_history` /
  `price_cache_metadata` / `position_snapshots` `WHERE instrument_id NOT
  IN (...)`. Bulk delete inside a single cache-side transaction; log
  the row count.
- Called from `run_boot_sequence` after migrations, before the FX/price
  refreshes.
- Covered by a test that deletes an instrument, runs the janitor, and
  asserts the cache rows are gone.

## 4. Part B — Cloud-sync-aware DB location

Goal: let the user point the **ledger** (and, optionally, the config)
file at a OneDrive / iCloud / Dropbox / Google Drive folder without
hand-editing config files, while keeping the cache strictly local.

### 4.1 Resolution order

For each of `ledger_path`, `config_path`, `cache_path` the resolver
walks:

1. Explicit env var: `INV_DASHBOARD_LEDGER_PATH` /
   `INV_DASHBOARD_CONFIG_PATH` / `INV_DASHBOARD_CACHE_PATH`.
2. Persisted value in `config.sqlite`'s `app_config`
   (`ledger_path` / `config_path`; **the cache path is never persisted
   to a synced file** — it lives only in env / user-data default).
3. First-run default, picked by `detect_cloud_sync_root()` — see §4.3.
4. Hard fallback: `$XDG_DATA_HOME/inv-dashboard/` on POSIX,
   `%LOCALAPPDATA%\inv-dashboard\` on Windows.

The cache always falls through to the platform's user-data dir.
Persisting it would risk a user accidentally sending GB of cached
prices through the cloud after a path edit.

### 4.2 `Settings` changes

- Replace `db_path` with `ledger_path`, `config_path`, `cache_path`
  (back-compat: if `INV_DASHBOARD_DB_PATH` is set and the new vars are
  not, treat it as the parent dir and derive the three).
- Add derived `ledger_url` / `config_url` / `cache_url` properties.
- Keep `_default_db_path()` as the hard-fallback root.

### 4.3 Cloud-folder detector

`detect_cloud_sync_root() -> Path | None` checks, in order:

- **OneDrive** — `%OneDrive%`, `%OneDriveConsumer%`,
  `%OneDriveCommercial%`; on POSIX, `~/OneDrive`.
- **iCloud** — `~/Library/Mobile Documents/com~apple~CloudDocs` (macOS),
  `%USERPROFILE%\iCloudDrive` (Windows).
- **Dropbox** — `~/Dropbox`, plus the `~/.dropbox/info.json` parser
  (handles relocated installs).
- **Google Drive** — `~/Google Drive`, `~/GoogleDrive`,
  `%USERPROFILE%\Google Drive`, plus the macOS
  `~/Library/CloudStorage/GoogleDrive-*` pattern.

Return the first existing directory or `None`. The result is suggested
in the onboarding wizard (Settings page also exposes a "Move ledger to
…" picker), but never silently applied: a first-run that finds OneDrive
shows a one-screen prompt "Store ledger in OneDrive\inv-dashboard?
[Yes / Choose another folder / Keep local]".

### 4.4 Settings UI

- New panel in `/settings` → "Storage":
  - Read-only display of resolved `ledger_path` / `config_path` /
    `cache_path` and which resolver step provided each.
  - "Move ledger…" button that opens a folder picker, copies the file
    via the existing `split_db`-style tool (atomic move + integrity
    check + rolling backup), updates `app_config`, and prompts for a
    restart.
  - Warning banner if the resolved ledger path is inside a known cloud
    folder but the file is not encrypted (see Part C) or sidecar files
    are present.

## 5. Part C — Security hardening

The threat model is: a laptop with the ledger file backed up to a
consumer cloud-drive folder, possibly shared family-of-devices, possibly
accessible to anyone who has the cloud account creds. We do **not**
defend against a privileged attacker on the running machine.

### 5.1 Encryption at rest for synced tiers

- Use **SQLCipher** for `ledger.sqlite` and `config.sqlite`. Driver
  swap is `sqlite+pysqlcipher://`; pragma listener gains
  `PRAGMA key = ?` as the first statement on each connect.
- Key material: derived via Argon2id from a user-supplied passphrase
  collected in the onboarding wizard, stored in the OS keychain
  (`keyring` lib: Windows Credential Manager / macOS Keychain /
  Secret Service on Linux). The passphrase is **not** stored on disk.
- A separate "recovery file" (the passphrase printed once, user-saved
  outside the cloud folder) is offered at setup time. We do not
  generate a key recovery escrow; loss of passphrase = loss of file.
- Cache stays unencrypted — it lives in the user-data dir, is
  regenerable, and SQLCipher would slow the per-symbol refresh loop
  unnecessarily.
- Migration step: the `split_db` tool gains `--encrypt-ledger` /
  `--encrypt-config` flags that wrap the copy step with `ATTACH
  DATABASE 'ledger.sqlite' AS plain KEY ''; ATTACH … AS enc KEY '…';
  SELECT sqlcipher_export('enc');`.
- Add `pysqlcipher3` (or `sqlcipher3-binary`) as an **optional** dep
  behind an extra (`pip install investment-dashboard[encrypted]`); the
  code path checks at boot and refuses to start if encryption is
  configured but the driver is missing.

### 5.2 No SQLite sidecar files in the cloud folder

WAL/SHM files (`-wal`, `-shm`) are per-connection state that cloud
sync engines will happily rewrite mid-write — the cleanest way to corrupt
a SQLite DB on OneDrive is to let the WAL sidecar replicate.

- For files in a detected cloud-sync folder, switch the journal mode
  from `WAL` to `TRUNCATE` (`PRAGMA journal_mode=TRUNCATE` in the
  pragma listener, gated on path). TRUNCATE keeps the journal inside a
  single file that is zero-length at rest and cleaned up on commit, so
  the cloud sync engine sees only `ledger.sqlite` changing.
- Cache stays on WAL (it's local; WAL is faster).
- Add a boot-time guard that scans the ledger/config dirs for stray
  `*.sqlite-wal` / `*.sqlite-shm` files; if found, log a loud warning
  and refuse to open until the user runs `inv-dashboard repair-sidecar`
  (which integrity-checks and removes them).
- Tell the user's cloud client to exclude the cache dir: documented in
  README, and the Settings → Storage panel surfaces the exact paths to
  add to OneDrive's "Always keep on this device" / Dropbox's selective
  sync exclusion.

### 5.3 Single-writer guard

Two laptops syncing the same `ledger.sqlite` on OneDrive will race; the
later writer overwrites the earlier with no merge.

- On startup, take an OS file lock (`fcntl.flock` on POSIX,
  `msvcrt.locking` on Windows) on `ledger.sqlite.lock` next to the
  ledger file. Lock file lives next to the DB and is itself excluded
  from sync.
- Lock acquisition is **non-blocking**; on failure, show a startup
  dialog "Another instance has the ledger open on this device or
  another synced device. Open read-only? [Read-only / Quit]".
- Read-only mode disables every write path (importer, manual entry,
  Settings edits) and surfaces a top-bar banner.
- The lock is released on graceful shutdown and on crash via OS
  cleanup; if a stale lock file is detected (lock acquirable but file
  present), it's overwritten silently.

### 5.4 Integrity & backups

- Boot runs `PRAGMA integrity_check` on the ledger and config files
  before opening sessions. On failure, the latest rolling backup is
  surfaced in a startup dialog with "Restore from backup" / "Quit"
  options.
- Rolling backups: on graceful shutdown and once per hour while
  running, snapshot the ledger and config via
  `sqlite3_backup_init` (`Connection.backup()` — SQLite-safe vs. a
  file copy) to `backups/ledger-YYYYMMDD-HHMMSS.sqlite`. Keep the last
  N=24 hourly, last 14 daily, last 12 monthly; delete the rest. Backups
  live next to the source and follow the same encryption + sidecar
  rules.
- An exportable "verified backup" command (`inv-dashboard backup
  --verify`) runs the integrity check on a fresh copy and writes a
  manifest with row counts per table — for the "I'm rotating laptops"
  case.

### 5.5 Audit & migration logging

- Every alembic migration logs `(file, from_rev, to_rev, applied_at)`
  to a `schema_audit` table on the same file. Useful when restoring an
  older backup and verifying it can be brought to head.
- The `split_db` and `repair-sidecar` tools append to a
  `tools_audit.jsonl` next to the ledger file with the operation, input
  hash, output hash, and counts.

## 7. Out of scope (explicit)

- **Live multi-device editing.** No cross-device write coordination
  ships in v2.0. The config-tier write queue table is *schema-ready* —
  a `pending_config_writes` table with `(id, device_id, applied_at,
  payload_json)` columns — but no consumer is wired in. A future v3
  can add a "secondary device" mode that records intents into that
  queue and lets the primary device replay them.
- **Server / hosted sync.** Still single-user, $0-host, runs on
  `0.0.0.0:8080` on the laptop.
- **Cross-tier joins.** The ORM already enforces this (`get_bind`
  routing in Phase 2 makes it explicit); services compose data from
  multiple sessions in Python.

## 8. Risks & open questions

- **SQLCipher on Windows wheel availability.** `pysqlcipher3` needs a
  pre-built wheel for the `py3.12-win_amd64` target; if upstream
  doesn't publish one, `sqlcipher3-binary` is the fallback. Validate
  before committing to Phase 4.
- **OneDrive "Files On-Demand"** can dehydrate the ledger file under
  memory pressure, breaking an open SQLite connection. Mitigation:
  Settings → Storage surfaces the "Always keep on this device" toggle
  with a direct deep-link; the boot integrity check catches the failure
  mode if a user ignores the prompt.
- **Atomic move across drives** in the Settings "Move ledger…" flow:
  fall back to copy-verify-delete with a temporary holdback file, not a
  bare `Path.rename`.
- **Test matrix.** Phase 2 doubles the SQL surface; reusing
  `ALL_METADATAS` plus a per-tier engine fixture in `tests/conftest.py`
  keeps test files unchanged. Plan to add an integration test
  (`tests/integration/test_split_layout.py`) that exercises the full
  three-engine boot path against `tmp_path` files.

